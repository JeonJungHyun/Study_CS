# ⚙️ 01. 운영체제

<details> <summary>1. 운영체제(OS)란?</summary> </br>

운영체제(OS, Operating System)는 하드웨어와 사용자 프로그램 사이에서 중간 역할을 수행하는 소프트웨어이다. </br>

우리가 흔히 알고 있는 윈도우(Windows), 맥OS(macOS), 리눅스(Linux) 와 같은 프로그램이 이에 해당한다. </br>

컴퓨터라는 하드웨어 상에 어플리케이션들이 동작되려면 이 하드웨어들과 적절하게 데이터틀 주고 받으며 논리적인 일들을 해야 하는데, </br>

이떼 운영체제가 사용자가 컴퓨터를 쉽게 다룰 수 있도록 사용자와 컴퓨터 사이에서 인터페이스를 제공하고, </br>

컴퓨터 시스템의 자원을 효율적으로 관리 및 분배하는 역할을 해주는 것이다. </br>

- 사용자 입장 : 프로그램 실행 가능
- 시스템 입장 : CPU, 메모리, 디스크, 입출력 장치 등 효율적인 관리 가능

</details>

<details> <summary>2. 운영체제(OS)가 필요한 이유</summary> </br>

✅ 사용자 편의 제공 </br>

- 운영체제가 없다면 사용자가 직접 하드웨어를 제거해야함 -> 매우 비효울적이다. </br>
  따라서 운영체제는 응용 프로그램이 하드웨어를 쉽게 사용할 수 있게 추상화된 환경을 제공한다. </br>

✅ 시스템 성능 향상 </br>

- 안정적인 시스템 운영 : 프로그램 오류가 전체 시스템에 영향을 덜 미치도록 해준다. </br>

- 보안 및 접근 제어 : 권한 없는 프로그램이 자원에 접근하지 못하게 해준다. </br>

- 자원 관리 : CPU, 메모리, I/O 등 시스템 자원을 효율적으로 배분해준다. </br>

- 프로세스 동시 실행 : 여러 프로그램을 동시에 실행 가능하게 해준다. </br>

</details>

<details> <summary>3. 프로세스(Process) / 스레드(Thread)</summary> </br>

1️⃣ 프로세스 </br>

1) 정의 </br>

프로세스는 현재 실행중인 프로그램을 의미한다. </br>
평소에는 보조 기억 장치인 디스크에 저장되어 있지만 사용자가 실행하면 디스크에 있던 프로그램이 메모리에 올라오고, </br>
메모리에 저장된 명령어가 CPU를 차지함으로써 실행할 수 있게 되는 구조이다. </br>

<img width="600" height="378" alt="image" src="https://github.com/user-attachments/assets/722462a6-c966-4b29-ac07-155ac9182012" /> </br>

2) 특징 </br>

- 각 프로세스는 독립된 메모리 공간(코드, 데이터, 스택)을 가진다. </br> 

- 다른 프로세스와 메모리를 공유하지 않는다. -> 안정성이 높다. </br>

3) 예시 </br>

- Chrome 브라우저 하나 → 하나의 프로세스
- Word 실행 → 하나의 프로세스 </br>

<img width="600" height="347" alt="image" src="https://github.com/user-attachments/assets/19143591-2c85-4107-be23-f109eaacc95c" /> </br>

🌟 여기서 알아둬야 할 점은, 프로그램은 하나이지만 프로세스는 여러 개일 수 있다는 것. </br>
예를 들어 구글 크롬의 경우 프로그램은 하나이지만 크롬 창을 여러 개 띄우고 동시에 작업할 수 있다. </br>
이는 크롬 프로그램이 메모리에 여러 번 올라와 여러 개의 프로세스로 존재하기 때문이다. </br>

2️⃣ 스레드 </br>

1) 정의 </br>

스레드(thread)는 프로세스 내에서 작업을 처리하는 실행 단위로, 프로세스가 할당받은 자원(CPU, 메모리 등)을 이용해 작업을 처리한다. </br>
즉, 한 프로세스 안에 여러개의 스레드가 생성될 수 있다. </br>

2) 특징 </br>

- **같은 프로세스 안의 스레드는 코드, 데이터, 파일 등 자원을 공유한다.** </br>

<img width="600" height="349" alt="image" src="https://github.com/user-attachments/assets/168cb1b5-d867-4455-b51d-8f3ef7c3f0e5" /> </br>

각각의 프로세스는 코드 영역, 데이터 영역, 힙 영역, 스택 영역으로 나뉘어 저장이 되는데, </br>
하나의 프로세스 내에 여러 개의 스레드가 있다면 각 스레드는 스택 영역만 따로 할당받고 나머지 영역을 공유한다. </br>
즉 프로세스 내 코드, 데이터, 파일 등 자원을 서로 공유하면서 실행되는 것이다. </br>

- **동시에 여러 작업을 수행할 수 있다.** </br>

하나의 프로세스를 여러 개의 스레드로 구성하고 둘 이상의 스레드가 동시에 작업을 처리하는 방식을 멀티스레드(multi-thread)라고 한다. </br>

멀티스레드는 작업을 분할해 동시에 처리함으로써 성능을 향상하고 자원을 효율적으로 사용할 수 있게 한다. </br>

하지만 여러 스레드가 자원을 공유하기 때문에 동기화 등의 문제가 발생할 수 있으므로 적절한 스레드 동기화 기술이 필요하다. </br>

- **스레드 간 통신이 빠르지만, 오류 발생 시 전체 프로세스 영향을 줄 수 있다.** </br>

프로세스 내 메모리 공간을 서로 공유하기 때문에 오류 발생 시 전체 영향을 줄 수 있다.

3) 예시 </br>

- Chrome 브라우저 탭마다 스레드 하나씩 → 같은 프로세스 내에서 공유 자원 사용 </br>

</details>

<details> <summary>3. 멀티프로세스(Multi-Process) / 멀티스레드(Multi-Thread)</summary> </br>

1️⃣ **멀티프로세스 (Multi-Process)** </br>

하나의 어플리케이션을 여러 개의 프로세스로 구성하여 각 프로세스가 하나의 작업(Task)을 처리하도록 하는 것. </br>

1. 장점 </br>

- 안정성 우수 </br>
부모 프로세스와 자식 프로세스는 각각 고유한 PID(Process ID)를 가지고 있으며, 독립적인 메모리 공간을 가지고 있어 서로 다른 작업을 수행한다. </br>
대표적인 예로 웹 브라우저의 상단 탭(Tab) 이나 새 창을 들 수 있다. </br> 
각 브라우저 탭은 같은 브라우저 프로그램 실행이지만, 각기 다른 사이트 실행을 행하기 때문이다. </br>
따라서 자식 프로세스 중 하나가 죽어도 다른 프로세스에 영향을 주지 않아 `전체 프로그램의 안정성이 높다`. </br>

- 구현의 단순성 및 보안성 </br>
각 프로세스는 자신만의 메모리 주소 공간을 가지기 때문에 한 프로세스가 다른 프로세스의 데이터를 함부로 열어보거나 수정할 수 없다. </br>
데이터를 공유하지 않는 구조 덕분에 여러 작업을 동시에 수행할 때 발생할 수 있는 데이터 오염 문제에서 비교적 자유롭고, </br>
보안적으로 더 안전하다는 특징이 있다. </br>

2. 단점 </br>

- Context Switching Overhead </br>
프로세스는 독립적인 메모리 영역을 가지기 때문에, CPU가 한 프로세스에서 다른 프로세스로 넘어갈 때 캐시 메모리에 있는 기존 데이터를 모두 비워야(Cache Flush) 한다. </br>
새로운 프로세스가 실행될 때마다 다시 데이터를 채워 넣어야 하므로, 전환 과정에서 CPU가 아무 일도 못 하는 '낭비되는 시간'이 매우 길어진다. </br>
결과적으로 시스템 전체의 처리 속도가 떨어진다. </br>

- 자원 소모 및 메모리 낭비 </br>
새로운 프로세스를 생성할 때마다 운영체제는 독립적인 메모리 영역(Code, Data, Stack, Heap)을 통째로 새로 할당해야 한다. </br>
똑같은 프로그램을 여러 개 실행하더라도 각각의 메모리 공간이 중복해서 필요하므로, 시스템 자원을 많이 잡아먹는 비효율적인 구조를 갖게 된다. </br>

2️⃣ **멀티스레드(Multi-Thread)</summary>** </br>

하나의 프로세스 내에서 여러 개의 실행 흐름(스레드)을 생성하여 작업을 처리하는 방식이다. </br>
스레드들은 프로세스의 자원을 공유하며 동시에 각기 다른 업무를 수행한다. </br>

1. 장점 </br>

- 자원의 효율성 </br>
멀티 스레드는 하나의 프로세스 내에서 여러 개의 스레드가 생성되며, 스레드들은 프로세스가 할당받은 메모리 중 Code, Data, Heap 영역을 공유한다. </br>
이를 통해, 프로세스 간 통신 (IPC)을 사용하지 않고도 데이터를 공유할 수 있기 때문에, 자원의 효율적인 활용이 가능해 시스템 자원 소모가 줄어든다. </br>

<img width="2040" height="920" alt="image" src="https://github.com/user-attachments/assets/91479a8d-aa06-4850-8d8c-3b969b96881d" />


- 강력한 응답성과 빠른 데이터 공유 </br>
스레드 간에는 힙(Heap) 영역을 통해 데이터를 즉시 주고받을 수 있다. </br>
또, 멀티프로세스처럼 복잡한 통신(IPC) 과정이 필요 없기 때문에 작업 간의 데이터 전달이 매우 빠르며, </br>
사용자의 요청에 즉각적으로 반응해야 하는 서비스(예: 게임, 채팅 앱 등)를 만들기에 매우 유리하다. </br>

- 컨텍스트 스위칭 비용 감소 </br>
스레드 간의 전환은 프로세스 간의 전환보다 훨씬 빠르다. </br>
같은 프로세스 내의 스레드들은 메모리 주소 체계를 공유하므로, CPU가 작업을 갈아탈 때 캐시 메모리를 비울 필요가 없기 때문이다. </br>
덕분에 전환 시 발생하는 오버헤드가 적어 시스템 전체의 처리량(Throughput)이 향상된다. </br>

2. 단점 </br>

- 안정성 취약 (연쇄 종료의 위험) </br>
멀티스레드 구조에서 모든 스레드는 하나의 프로세스라는 '운명 공동체'이다. </br>
만약 특정 스레드에서 치명적인 오류(예: 잘못된 메모리 참조)가 발생하여 프로세스가 강제 종료되면, </br>
그 안에서 실행 중이던 모든 스레드가 한꺼번에 중단된다. </br>
이는 멀티프로세스가 가진 '독립적인 안정성'과는 대조되는 가장 큰 약점이다. </br>

<img width="2220" height="1444" alt="image" src="https://github.com/user-attachments/assets/240dceb2-6887-4687-86aa-7d7284194902" />

- 동기화로 인한 성능 저하 </br>
멀티 스레드 모델은 여러 개의 스레드가 공유 자원에 동시에 접근할 수 있기 때문에, 동기화 문제가 발생할 수 있다. </br>
여러 스레드가 공유 자원(데이터)에 동시에 접근하여 값을 수정하려고 한다면, 의도되지 않은 엉뚱한 값을 읽어 서비스에 치명적인 버그가 생길수도 있다. </br>
따라서 스레드 간 동기화(syncronized)는 데이터 접근을 제어하기 위한 필수적인 기술이다. </br>
동기화 작업은 여러 스레드들이 자원에 대한 접근을 순차적으로 통제하는 것이다. </br>
그러면 동시 접근으로 인한 동시 수정과 같은 현상은 일어나지 않게 된다. </br>

- 교착 상태(Deadlock)의 위험 </br>
데이터 꼬임을 막기 위해 공유 자원에 '자물쇠(Lock)'를 거는 처리가 필요한데, </br>
이 과정에서 서로가 가진 자물쇠가 풀리기를 무한정 기다리는 **'교착 상태(Deadlock)'**에 빠질 수 있다. </br>
이를 해결하기 위해 설계가 매우 복잡해지며, 잘못 설계할 경우 오히려 성능이 크게 떨어질 수 있다. </br>
