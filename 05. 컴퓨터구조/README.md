# 🧬 05. 컴퓨터 구조 (Architecture) 
<details> <summary>1. 컴퓨터 구조: 폰 노이만 구조와 하드웨어의 상호작용</summary> </br>
  
모든 현대 컴퓨터는 `프로그램 내장 방식`을 제안한 폰 노이만의 설계를 따른다. </br>
이는 실행하려는 프로그램의 명령어와 데이터를 반드시 메모리에 먼저 저장해두어야 한다는 원칙이다. </br>
컴퓨터의 구조는 크게 CPU / 메모리 / 보조기억장치 / 입출력장치로 나눌 수 있다. </br>

</br>

❶ CPU (Central Processing Unit) </br>
- `중앙처리장치` </br>

- 메모리에 저장된 명령어를 읽어 들이고, 읽어 들인 명령어를 해석하고 실행한다. </br>

</br>

❷ 메모리 (RAM) </br>
- `주기억장치` </br>
- 현재 실행되는 프로그램 (= 프로세스)의 `명령어와 데이터를 저장`하는 부품 </br>
- 프로그램이 실행되려면 반드시 해당 프로그램의 데이터와 명령어가 메모리에 저장되어 있어야 한다. </br>
  🌟 다만, 운영체제의 `페이징 (Paging)`을 다루면 실행되는 프로그램의 데이터와 명령어를 절약할 수 있다. </br>

- 저장된 `명령어와 데이터의 위치가 정돈`되어 있어야 빠르게 접근할 수 있기 때문에, `주소`라는 개념이 사용된다. </br>

</br>

❸ 보조기억장치 (SSD/HDD) </br>
- 전원이 꺼져도 보관해야 할 데이터를 저장하는 `거대 창고` </br>

- 메모리는 가격이 비싸 저장 용량이 적다. 또한, 전원이 꺼지면 저장된 내용을 잃는다는 단점이 있다. (휘발성) </br>
보조기억장치는 이런 문제를 해결해 주는 저장 장치이다. 하드디스크가 대표적인 보조기억장치의 예다. </br>
`메모리`는 실행할 정보를, `보조기억장치`는 보관할 정보를 저장한다. </br>

</br>

❹ 입출력장치 </br>

입출력장치는 `컴퓨터 외부에 연결되어 컴퓨터 내부와 정보를 교환`하는 장치 </br>

✏️ 사실 하드디스크, SSD 등도 컴퓨터 외부에 연결되긴 하지만, 이들은 입출력 기능에 비해 </br>
메모리를 보조하는 것이 더 주된 기능이다 보니 보조기억장치로 분류되곤 한다. </br>

</br>

🚌 시스템 버스와 메인 보드 </br>

위의 **컴퓨터 부품들이 연결될 수 있는 판**을 `메인 보드`라 하며, 이 컴퓨터 부품들은 `버스 (bus)`를 이용해 서로 정보를 주고받을 수 있다. </br>
그리고 버스 중 가장 중요한, 즉 **컴퓨터의 네 가지 핵심 부품을 연결하는 가장 중요한 버스** 를 `시스템 버스`라 한다. </br>
  
</details>

<details> <summary>2. CPU의 동작 원리와 명령어 사이클</summary> </br>

CPU는 단순히 계산만 하는 게 아니라, 명령어를 정해진 순서대로 처리하는 정교한 부품들의 집합이다. </br>
CPU가 명령어를 처리하는 정형화된 흐름을 명령어 사이클이라 한다. </br>

</br>

❶ ALU (Arithmetic Logic Unit): 산술논리연산장치
- 실제로 덧셈, 뺄셈, 논리 연산을 수행하는 `계산기` </br>

- 컴퓨터 내부에서 수행되는 대부분의 계산을 처리한다. </br>

</br>

❷ 레지스터 (Register): CPU 내부의 초고속 임시 저장소
- CPU 내부에 있는 작은 임시 저장 장치. 프로그램을 실행하는 데 필요한 값들을 임시로 저장한다. </br>

- CPU 안에는 여러 개의 레지스터가 존재하며, 각기 다른 이름과 역할을 가지고 있다. </br>
  - **프로그램 카운터**: 다음번에 실행할 명령어의 주소를 저장한다. </br>
  
  - **명령어 레지스터**: 메모리에서 읽어 들인 명령어의 주소를 저장한다. </br>
  
  - **플래그 레지스터**: ALU 연산 결과, CPU 상태에 대한 부가적인 정보를 저장한다. </br>

</br>

❸ CU (Control Unit): 제어 장치 </br>
- 제어 신호 (control signal)를 내보내고, 명령어를 해석한다. </br>

- 제어 신호는 컴퓨터 부품들을 관리하고 작동시키기 위한 일종의 전기 신호니다. (메모리 읽기, 쓰기 등)

</br>


♻️ CPU의 명령어 사이클 </br>

1. 인출 사이클 </br>
우선 CPU는 메모리로부터 명령어를 가져온다. 메모리에는 현재 실행되는 프로그램의 명령어와 데이터가 보관되어 있기 때문이다. </br>

2. 해석 </br>
이게 무슨 명령인지 제어장치가 분석한다. </br>

3. 실행 사이클 </br>
가져온 명령어를 실행한다. </br>

이렇게 단순히 인출 사이클과 실행 사이클을 반복하여 명령어 사이클이 이뤄진 경우도 있지만, </br>
메모리 접근이 추가로 필요한 상황이 발생하기도 하며 (명령어의 주소 지정 방식 등에 따라 달라짐), `인터럽트`가 발생하는 경우도 있다. </br>

</details>

<details> <summary>3. 효율적인 업무 처리: 인터럽트(Interrupt)와 DMA</summary> </br>

✏️ 인터럽트란 ? </br>
`CPU의 작업을 방해하는 신호`이다. </br>

인터럽트가 발생하면, `CPU는 자신이 하던 일을 잠시 중단하고 다른 일을 수행`하게 된다. </br>

</br>

❶ 동기 인터럽트 (예외, Exception) </br>

동기 인터럽트는 `CPU에 의해 발생하는 인터럽트`이다. </br>
CPU가 `명령어를 수행하는 도중 예상치 못한 상황에 마주하거나`, `프로그래밍 오류와 같은 예외적인 상황에 마주쳤을 때` 인터럽트가 발생할 수 있다. </br>

동기 인터럽트의 종류는 아래와 같다. </br>

- 폴트(Fault): 수습 가능한 오류이다. 문제를 해결한 뒤, 에러가 났던 그 명령어부터 다시 시작한다. </br>
    (예: 없는 데이터를 찾았을 때 메모리에서 가져온 후 재실행) </br>

- 트랩(Trap): 의도적인 호출이다. 처리가 끝나면 그다음 명령어부터 실행한다. </br>
    주로 디버깅이나 시스템 콜(System Call)에서 쓰인다. </br>

- 중단: 복구 불가능한 심각한 오류이다. </br>
    CPU가 실행 중인 프로그램을 강제로 중단시킬 수밖에 없는 심각한 오류를 발견했을 때 발생한다. </br>

- 소프트웨어 인터럽트: 사용자가 운영체제에 도움을 요청하는 '시스템 콜' 시 발생한다. </br>

</br>

❷ 비동기 인터럽트 (하드웨어 인터럽트) </br>

주로 `입출력 장치(마우스, 키보드, 프린터)` 가 보내는 알림이다. CPU가 언제 올지 예측할 수 없어서 '비동기'라고 부른다. </br>
예시로 CPU가 프린터와 같은 입출력장치에 입출력 작업을 부탁하면, 작업을 끝낸 입출력장치가 CPU에게 완료되었다는 알림을 보낸다. </br>
만약 CPU가 프린터의 작업이 끝날 때까지 계속 완료되었는지를 물어본다면 CPU를 효율적으로 쓸 수 없을 것이다. </br>
이렇게 `입출력장치가 알림을 보내기 전까지 CPU가 다른 일을 수행`할 수 있다 보니, 동기 인터럽트와 다르게 비동기 인터럽트라 칭하는 것이다. </br>

</br>

> **폴링(Polling) vs 인터럽트** </br>
>
> 폴링: CPU가 장치에게 입출력 장치의 상태는 어떤지, 처리할 데이터가 있는지 주기적으로 확인하는 방식 (CPU 낭비 심함). </br>
> 인터럽트: 장치가 다 되면 CPU에게 "다 됐어!"라고 소리치는 방식 (효율적). </br>

</details>

<details> <summary>4. 인터럽트 처리 순서: "어떻게 복구하는가?"</summary> </br>

❶ 인터럽트 확인: CPU는 하나의 명령어를 끝낼 때마다 "혹시 들어온 신호가 있나?" 확인. </br>

❷ 플래그 체크: CPU의 플래그 레지스터를 보고 지금 인터럽트를 받아줄 수 있는 상태인지 확인. </br>
(절대 방해받으면 안 되는 중요한 일 중에는 인터럽트를 막아두기도 한다.) </br>

  - CPU가 인터럽트 요청을 수용하기 위해서는 플래그 레지스터의 `인터럽트 플래그`가 활성화되어 있어야 한다. </br>
  
  - 특히 하드웨어 인터럽트의 경우에는 막을 수 있는 인터럽트(maskable interrupt)와 막을 수 없는 인터럽트(non maskable interrupt)가 있다. </br>

❸ 작업 백업: 인터럽트를 받아들이기로 했다면, 나중에 돌아와서 다시 일할 수 있도록 현재까지의 작업 상태(레지스터 값 등)를 스택 메모리에 안전하게 저장. </br>

❹ 서비스 루틴 실행: `인터럽트 벡터(지도)`를 보고 해당 인터럽트를 처리할 전용 프로그램인 `인터럽트 서비스 루틴(ISR)`을 찾아가서 실행. </br>

  - 인터럽트 서비스 루틴은 인터럽트를 처리하기 위한 프로그램이다. </br>
  
  - 인터럽트 벡터는 서로 다른 여러 개의 인터럽트 서비스 루틴을 식별하기 위한 정보이다. </br>

❺ 복구 및 재개: 처리가 끝나면 3번에서 저장해둔 데이터를 다시 불러와서 멈췄던 지점부터 다시 일을 시작. </br>

</details>

<details> <summary>5. 입출력 장치의 연결: 장치 컨트롤러와 드라이버</summary> </br>

CPU와 메모리는 엄청나게 빠르지만, 키보드나 마우스는 아주 느리다. </br>
이 둘을 직접 연결하면 속도 차이 때문에 CPU가 계속 기다려야 하는 비효율이 발생한다. 그래서 중간 관리자를 둔다. </br>

</br>

❶ 장치 컨트롤러 (Hardware)

입출력 장치와 컴퓨터 본체를 연결하는 하드웨어 칩. </br>

🤔 왜 필요할까? </br>

- 정보 규격화: 세상의 수많은 마우스, 키보드의 신호를 CPU가 알아듣는 공통 언어로 `번역`해주며, </br>
입출력장치가 가졌던 규격화의 어려움을 해결할 수 있다. 이 과정에서 `오류를 검출`하기도 한다. </br>

- 데이터 버퍼링: 빠른 CPU와 느린 장치 사이에서 데이터를 잠시 모았다가 한꺼번에 보내서 속도를 맞춘다. </br>

</br>

✏️ 장치 컨트롤러의 구조: 레지스터 </br>
장치 컨트롤러의 대략적인 구조는 데이터 레지스터, 상태 레지스터, 제어 레지스터가 있다. </br>

- 데이터 레지스터: CPU와 입출력장치 사이에 주고받을 `데이터가 담기는 레지스터(버퍼)`. </br>

- 상태 레지스터: 입출력장치가 입출력 작업을 할 준비가 되었는지, 입출력 작업이 완료되었는지 등의 `상태를 저장`. </br>

- 제어 레지스터: 입출력장치가 `수행할 내용에 대한 제어 정보와 명령을 저장`. </br>

</br>

❷ 장치 드라이버 (Software) </br>

장치 컨트롤러가 입출력장치를 연결하기 위한 하드웨어적인 통로라면, 장치 드라이버는 `입출력장치를 연결하기 위한 소프트웨어적 통로`. </br>
장치 컨트롤러의 동작을 감지하고 제어함으로써 장치 컨트롤러가 컴퓨터 내부와 정보를 주고받을 수 있도록 할 수 있는 프로그램이며, </br>
하드웨어가 아니라 소프트웨어이므로 메모리에 저장된다. </br>

- 역할: 운영체제가 "프린트해!"라고 시키면, 드라이버가 그걸 받아서 해당 장치 컨트롤러가 이해할 수 있는 신호로 바꿔 전달. </br>

💡 실생활 비유:
요리사(CPU)가 외국인 보조(입출력 장치)에게 일을 시킬 때, 중간에 통역사(장치 드라이버)와 무전기(장치 컨트롤러)를 사용하는 것과 같다. </br>
무전기에는 메시지가 쌓이고(버퍼), 통역사는 요리사의 말을 외국인이 알아듣게 바꿔준다. </br>

</details>

<details> <summary>6. 데이터를 주고받는 3가지 방법</summary> </br>

❶ 프로그램 입출력 (Programmed I/O) </br>

CPU가 직접 명령어를 통해 장치 컨트롤러의 레지스터를 일일이 확인하고 데이터를 옮기는 방식. </br>


- 프로그램 속 `명령어로 입출력장치를 제어하는 방법`이다. </br>

- CPU가 프로그램 속 명령어를 실행하는 과정에서 입출력 명령어를 만나면 CPU는 입출력 장치에 연결된 </br>
장치 컨트롤러와 상호작용하며 입출력 작업을 수행한다.

- CPU가 장치 컨트롤러의 레지스터 값을 읽고 씀으로써 이루어집니다. 이때 장치 컨트롤러의 레지스터 위치는 </br>
`메모리 맵 입출력, 고립형 입출력 방식`으로 저장될 수 있다.

</br>

👎 단점: CPU가 장치가 다 끝날 때까지 계속 확인해야 해서 다른 일을 못 한다. </br>

</br>

❷ 인터럽트 기반 입출력 (Interrupt-driven I/O) </br>

장치 컨트롤러가 일을 끝내면 CPU에게 "다 됐어요!"라고 신호를 보내는 방식. </br>

- 인터럽트 기반의 입출력이다. </br>

- 장치 컨트롤러가 입출력 작업을 끝낸 뒤 CPU에게 인터럽트 요청 신호를 보내면, </br>
CPU는 하던 일을 잠시 백업하고 인터럽트 서비스 루틴을 실행한다. </br>

- 여러 개의 입출력장치에서 **동시다발적으로 인터럽트가 발생**하는 경우, `PIC (Programmable Interrupt Controller)`를 이용하여 </br>
**우선순위를 판단**한 뒤 가장 먼저 처리해야 할 인터럽트를 찾고 처리한다. </br>

</br>

❸ DMA (Direct Memory Access) </br>

데이터 옮기는 일만 전담하는 **대리인(DMA 컨트롤러)** 을 두는 방식. </br>

- 프로그램 입출력과 인터럽트 기반 입출력은 모두 CPU와 직접 상호작용한다. </br>
이는 하드 디스크 백업 등 CPU 부담이 매우 크다면 비효율적이다. </br>
  
- DMA 입출력 방식은 CPU를 직접 거치지 않고 DMA (Direct Memory Access)를 이용하여 </br>
입출력장치와 메모리가 CPU를 거치지 않고도 상호작용 할 수 있도록 하는 방식이다. </br>

- DMA 입출력 방식은 시스템 버스에 연결된 DMA 컨트롤러가 필요하다. </br>

- CPU는 이제 입출력의 시작과 끝에만 관여한다. </br>

- 그러나 시스템 버스는 CPU와 DMA 컨트롤러가 동시에 사용할 수 없기 때문에, </br>
DMA 컨트롤러는 CPU가 시스템 버스를 이용하지 않을 때마다 조금씩 시스템 버스를 이용하거나 </br>
CPU가 일시적으로 시스템 버스를 이용하지 않도록 허락을 구하고 시스템 버스를 이용한다. </br>
(이를 사이클 스틸링이라고도 합한다.) </br>

</br>

👍 결과: CPU의 업무 효율이 극대화된다. </br>

</details>

<details> <summary>7. 명령어 성능 향상 기법: 파이프라이닝 (Pipelining)</summary> </br>

CPU가 명령어를 하나씩 순차적으로만 처리하면 비효율적이다. 공장의 컨베이어 벨트처럼 여러 명령어를 겹쳐서 처리하는 기술이 필요하다. </br>

</br>

❶ 파이프라이닝 (Pipelining)

하나의 명령어가 인출 -> 해석 -> 실행 단계를 거치는 동안, 다음 명령어를 미리 인출하여 실행 단계를 겹치는 방식이다. </br>
이론적으로는 단계 수만큼 속도가 빨라지지만, 흐름이 끊기는 **해저드(Hazard)**가 발생할 수 있다. </br>

</br>

❷ 파이프라인 해저드 (Hazard)

- 데이터 해저드: 앞선 계산 결과가 나와야 다음 계산이 가능한 경우. </br>
- 제어 해저드: if문 같은 조건문 때문에 다음에 실행할 명령어가 바뀔 수 있는 경우. </br>

이를 해결하기 위해 CPU는 **분기 예측(Branch Prediction)**을 통해 다음에 올 명령어를 미리 짐작해서 실행한다. </br>

</details>

<details> <summary>8. 메모리 계층 구조와 캐시(Cache) 전략</summary> </br>

CPU의 연산 속도와 메모리의 접근 속도 차이(Memory Wall)를 극복하기 위한 구조이다. </br>

</br>
 
❶ 캐시 메모리 (Cache Memory) </br>

CPU와 RAM 사이의 속도 차이를 줄이기 위해 CPU 근처에 배치한 고속 메모리. </br>
참조 지역성(Locality) 원리에 따라 자주 쓰일 법한 데이터를 미리 가져다 놓는다. </br>

- 시간 지역성: 최근 접근한 데이터에 또 접근할 확률이 높음 (예: 반복문의 변수). </br>

- 공간 지역성: 접근한 데이터 주변의 데이터에 접근할 확률이 높음 (예: 배열의 순차적 탐색). </br>

</br>

❷ 가상 메모리 (Virtual Memory) </br>

실제 RAM 용량보다 큰 프로그램을 실행하기 위해 보조기억장치(SSD)의 일부를 메모리처럼 사용하는 기술. </br>

- LRU (Least Recently Used) 알고리즘: 메모리가 꽉 찼을 때, 가장 오랫동안 참조되지 않은 페이지를 교체 대상으로 삼는 전략. (가장 효율적인 방식!) </br>

</details>
