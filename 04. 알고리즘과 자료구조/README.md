# 🧠 04. 알고리즘과 자료구조

<details> <summary>1. 알고리즘(Algorithm)이란?</summary> </br>
❶ 알고리즘이란? </br>
문제를 해결하기 위한 절차 또는 방법이다. </br>
즉, 입력을 받아 원하는 출력 결과를 만들어내는 일련의 과정이다. </br>
예를 들어 라면을 끓이는 과정도 하나의 알고리즘이다. </br>
물을 끓인다 -> 면과 스프를 넣는다 -> 일정 시간 후 완성 </br>
🌟 이렇게 단계적으로 문제를 해결하는 방식이 알고리즘이다. </br>
</br>

❷ 알고리즘의 구성 요소 </br>
입력(Input) : 문제 해결을 위한 데이터 </br>
출력(Output) : 결과값 </br>
처리 과정(Process) : 문제 해결 절차 </br>

❸ 알고리즘의 조건 </br>
명확성 : 단계가 분명해야 한다 </br>
유한성 : 반드시 종료되어야 한다 </br>
효율성 : 가능한 빠르게 수행되어야 한다 </br>
입력 : 0개 이상의 입력 존재 </br>
출력 : 최소 1개 이상의 출력 존재 </br>
</details>

<details> <summary>2. 자료구조(Data Structure)란?</summary> </br>
❶ 자료구조란? </br>
데이터를 효율적으로 저장하고 관리하기 위한 구조이다. </br>
알고리즘이 “어떻게 풀까”라면 자료구조는 “어떻게 담을까”이다. </br>
  
</br>

❷ 왜 중요한가? </br>
같은 데이터를 사용하더라도
어떤 자료구조를 쓰느냐에 따라 성능이 달라진다. </br>
예를 들어, </br>
배열 : 접근 빠름 </br>
연결 리스트 : 삽입/삭제 빠름 </br>
🌟 따라서 문제에 맞는 자료구조 선택이 중요하다. </br>
</details>

<details> <summary>3. 자료구조의 분류</summary> </br>
자료구조는 크게 네 가지로 나뉜다. </br>

</br>

❶ 단순 구조 (Simple Structure) </br>
더 이상 분해할 수 없는 기본 자료형으로, 하나의 값만 저장하는 구조이다. </br>
`정수형, 실수형, 문자형, 논리형` 등이 이에 해당한다. </br>

</br>

❷ 선형 구조 (Linear Structure) </br>
데이터가 일렬로 순차적으로 연결된 구조로, 각 요소가 앞뒤로 하나씩만 연결된다. </br>
`배열, 연결 리스트, 스택, 큐, 덱` 등이 이에 해당한다. </br>

</br>

❸ 비선형 구조 (Non-Linear Structure) </br>
`데이터가 계층적 또는 복잡한 관계로 연결된 구조`로, 하나의 요소가 여러 요소와 연결될 수 있다. </br>
트리와 그래프가 대표적인 예이다. </br>
</br>

❹ 파일 구조 (File Structure) </br>
데이터를 보조기억장치에 저장하기 위한 구조로, `대용량 데이터를 효율적으로 관리하기 위해 사용된다.` </br>
`순차 파일, 직접 파일, 색인 파일` 등이 이에 해당한다. </br>

</details>

<details> <summary>4. 시간 복잡도 (Time Complexity)</summary> </br>

❶ 시간 복잡도란? </br>
알고리즘의 성능을 나타내는 기준. 단순히 "몇 초 걸린다"가 아니라, `입력되는 데이터 양(N)에 따라 연산 횟수가 얼마나 늘어나는가`를 측정한다. </br>

❷ Big-O 표기법 (O) </br>
알고리즘의 효율성을 상한선(최악의 경우) 기준으로 표기하는 약속. </br>

- O(1) - 상수 시간: </br>
데이터가 1개든 100만 개든 연산 횟수가 똑같다. (가장 빠름) </br>

- O(logN) - 로그 시간: </br>
데이터가 2배 늘어날 때 연산은 1번만 늘어난다. (이진 탐색) </br>
상대가 생각한 숫자를 맞추기 위해서는 물론 무작위 숫자를 마구잡이로 불러서 맞춰 버리거나 아예 1부터 불러서 맞출 수도 있을 것이다. </br>
하지만 범위가 10, 100 이런 것이 아니라 몇 만, 몇 억이 된다면 그만큼 굉장한 시간 걸릴 것이기 때문에 더 효율적인 방법은 만약 범위가</br>
1~100인 숫자를 생각했다면 절반씩 나누어 불러서 그 숫자가 포함되는 범위를 좁혀나가면서 찾는 것이 가장 최선일 것이다.
보통 한 번의 연산이 진행될 때마다 연산의 범위가 반 씩 줄어들기 때문에 이를 O(logN)이라고 할 수 있다.

- O(N) - 선형 시간: </br>
입력 데이터의 크기에 비례해서 시간이 소요되는 알고리즘. (단순 반복문) </br>

`
for i in range(N):
    #do something`

위의 for문이 대표적인 O(N)의 알고리즘으로 만약 N의 크기가 10이면 10번, 100이면 100번 만큼 순회의 과정을 거치게 되어 </br>
그(N)만큼의 시간이 소요가 될 것이다. </br>

- O(N2) - 제곱 시간: </br>
입력 데이터의 제곱에 비례해서 시간이 소요가 되는 알고리즘. (이중 반복문) </br>

`
for i in range(N): 
    for j in range(N):
        #do something`

위처럼 이중 for문의 형태가 그 예시이다. </br>
N번 순회하는 것이 두 번 중첩되어 제곱의 형태로 시간이 소요 되는 결과가 나오는 것. </br>

❸ 왜 중요한가? (실무 포인트) </br>
백준 문제나 실무 서버 환경에는 보통 시간 제한이 있다. </br>
데이터(N)가 100만 개일 때, O(N) 알고리즘은 0.01초면 끝나지만 O(N2) 알고리즘은 2시간이 넘게 걸릴 수 있다. </br>
"내 코드가 서버를 터뜨릴지 아닐지"를 미리 계산하는 도구다. </br>

</details>

<details> <summary>5. 선형 구조 ① : 배열과 연결 리스트 </summary> </br>
둘 다 데이터를 일렬로 세우지만, 메모리에 저장하는 방식이 다르다. 이 차이 때문에 용도가 갈린다. </br>

</br>

❶ 배열 (Array): "영화관 지정석" </br>

- 저장 방식: 메모리의 연속된 공간에 데이터를 다닥다닥 붙여서 저장한다. </br>

- 장점: 인덱스(번호)만 알면 바로 찾아갈 수 있습니다. 조회(O(1))가 매우 빠르다. </br>

- 단점: 중간에 누가 끼어들려면 뒤 사람들이 다 한 칸씩 옆으로 밀려나야 한다. 삽입/삭제(O(N))가 느리다. </br>

</br>

❷ 연결 리스트 (Linked List): "기차 놀이" </br>

- 저장 방식: 데이터들이 메모리 여기저기 흩어져 있고, 앞사람이 뒷사람의 손(주소)을 잡고 연결된다. </br>

- 장점: 중간에 누가 끼어들면 손을 풀고 새 사람과 잡기만 하면 됩니다. 삽입/삭제(O(1))가 빠르다. </br>

- 단점: 특정 사람을 찾으려면 맨 앞부터 하나씩 세면서 가야 합니다. 조회(O(N))가 느리다. </br>

<details> <summary>6. 선형 구조 ② : 스택(Stack)과 큐(Queue)</summary> </br>
데이터를 넣고 빼는 순서가 핵심인 자료구조이다. </br>

</br>

❶ 스택 (Stack): "프링글스 통" </br>

- 원리: LIFO (Last In First Out) - 나중에 들어온 게 먼저 나간다. </br>

- 연산: push(넣기), pop(빼기). </br>
흔히 스택은 접시쌓기를 생각하면 이해하기 편하다. 가장 마지막에 내려놓은 접시를 가장 먼저 빼야지 쌓아놓은 접시들이 무너지지 않는다. </br>
이때, 접시(데이터)를 넣는 작업을 Push, 접시를 빼는 작업을 Pop이라고 부른다. </br>
또한 스택은 완전히 꽉 찼을 때 Overflow상태라고 하고, 완전히 비어 있으면 Underflow상태라고 한다. </br>

<img width="1000" height="394" alt="image" src="https://github.com/user-attachments/assets/a02fc2a0-a4fc-48c5-a33d-cdde2bd0f944" />

</br>

- 활용: 브라우저 뒤로 가기, 실행 취소(Ctrl+Z), 함수의 호출 기록. </br>

<img width="479" height="432" alt="image" src="https://github.com/user-attachments/assets/ee804b81-55d6-44c4-9efc-602b321e23ca" />

- 왜 쓰는가? </br>
데이터가 들어온 역순으로 다시 꺼내서 작업해야 할 때 사용. </br>

- 함수 호출 기록 (Call Stack): 프로그램에서 함수 A가 함수 B를 부르면, B가 끝날 때까지 A를 잠시 스택에 쌓아둔다. </br>

- 실행 취소 (Undo / Ctrl+Z): 가장 마지막에 한 작업부터 되돌려야 하므로 스택을 쓴다. </br>

- 뒤로 가기: 브라우저에서 방금 본 페이지를 스택에 쌓아두면 '뒤로 가기' 버튼을 눌렀을 때 가장 최근 페이지가 나온다. </br>

- 수식 계산 및 괄호 검사: (( )) 처럼 괄호가 짝이 맞는지 확인할 때 쓴다. </br>

</br>

❷ 큐 (Queue): "편의점 음료수 진열대" </br>

- 원리: FIFO (First In First Out) - 먼저 들어온 게 먼저 나간다. </br>

- 연산: enqueue(넣기), dequeue(빼기). </br>

- 활용: 은행 창구 대기열, 프린터 인쇄 대기열, 서버의 작업 처리 대기. </br>

</details>

<details> <summary>7. 해시 테이블 (Hash Table)</summary> </br>

❶ 해시 테이블의 정의 </br>
키(Key)를 해시 함수(Hash Function)라는 계산기에 넣어 나온 결과값(Hash Code)을 주소로 사용하여 데이터를 저장하는 자료구조. </br>

- 핵심: "이름이 뭐야?"라고 물었을 때, 1번부터 찾는 게 아니라 이름 그 자체가 주소가 되어 즉시 찾아간다. </br>

❷ 주요 특징과 성능 </br>

- 시간 복잡도: 조회, 삽입, 삭제 모두 O(1)이다. 데이터가 1억 개라도 키만 알면 한 번에 찾는다. </br>

- 공간 복잡도: 빠른 속도를 위해 미리 메모리 공간을 많이 확보해 두어야 하므로 공간을 많이 차지한다. (속도를 얻고 메모리를 내어주는 방식) </br>

❸ 해시 충돌 (Hash Collision) </br>
서로 다른 키를 넣었는데 해시 함수가 똑같은 번호(주소)를 뱉어내는 상황. (예: '정현'과 '철수'가 둘 다 3번 사물함을 배정받음) </br>

- 해결법 1. 체이닝 (Chaining): 같은 번호에 여러 데이터가 들어오면 연결 리스트로 줄을 세운다. (가장 많이 쓰임) </br>

- 해결법 2. 개방 주소법 (Open Addressing): 이미 주인이 있다면 비어 있는 다른 번호를 찾아 떠난다. </br>

❹ 백엔드 실무 활용 </br>

- Redis (인메모리 DB): 모든 데이터를 키-값(Key-Value) 형태로 저장하는 해시 테이블 기반 시스템. </br>

- 세션(Session) 관리: 로그인한 사용자의 정보를 토큰(Key)으로 즉시 조회할 때 쓴다. </br>

- 중복 체크: 수만 개의 데이터 중 "이미 등록된 이메일인가?"를 확인할 때 쓴다. </br>

</details>

<details> <summary>8. 트리 (Tree) 와 데이터베이스 인덱스</summary> </br>

❶ 트리의 정의 </br>
데이터를 계층적으로 연결한 구조. 거꾸로 세워진 나무 모양이며, 최상위 노드를 루트(Root), 자식이 없는 끝 노드를 **리프(Leaf)**라고 부른다. </br>

❷ 이진 탐색 트리 (Binary Search Tree, BST) </br>

- 규칙: 왼쪽 자식은 나보다 작고, 오른쪽 자식은 나보다 큰 값을 가진다. </br>

- 성능: 데이터를 찾을 때마다 탐색 범위가 절반으로 줄어들어 O(logN)의 속도를 보장한다. </br>

- 문제점: 데이터가 한쪽으로만 쏠리면(예: 1, 2, 3, 4 순서로 들어옴) 일렬로 늘어져서 성능이 배열처럼 느려질 수 있다. </br>

❸ B-Tree (비-트리) </br>
이진 탐색 트리의 단점을 보완하여, 스스로 균형을 잡는 트리. </br>

- DB 인덱스의 비밀: 우리가 사용하는 MySQL, Oracle 등 대부분의 DB 인덱스가 이 B-Tree 구조로 설계되어 있다. </br>

- 왜 쓰는가?: 하드디스크(보조기억장치)는 접근이 느리다. B-Tree는 한 번의 접근으로 많은 데이터를 읽을 수 있게 설계되어 DB 성능을 극대화한다. </br>

❹ 실무 및 용어 </br>

- 깊이(Depth): 루트에서 특정 노드까지 가는 경로의 길이. </br>

- 트래버설(Traversal): 트리 전체를 훑는 방법 (전위, 중위, 후위 순회). </br>

- 활용: 파일 시스템 구조(폴더 안의 폴더), 조직도, HTML의 DOM 구조. </br>

</details>
